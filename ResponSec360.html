ResponSec360 v1.0.0

A lightweight, modular log correlation engine built in Python — designed to simulate real SOC-level detection across Linux auth logs, Apache access logs, and syslog sources.

Python 3.8+ 10 Detection Rules MITRE ATT&CK Aligned MIT License

Overview

ResponSec360 is a detection engineering project that demonstrates the core concepts behind enterprise SIEM systems — log ingestion, normalization, rule-based correlation, and alerting — without requiring a full Elastic or Splunk stack.

It is designed to be educational and practical: the code is structured the same way a production detection platform would be, with clean separation between ingestion, parsing, correlation, and alerting layers.

What ResponSec360 does

CapabilityDetail
Log ingestionTails live log files or replays historical logs. Supports multiple simultaneous sources via threaded workers.
NormalizationEvery log line is parsed into a structured LogEvent (inspired by Elastic Common Schema) before detection runs.
Correlation10 detection rules use time-windowed state machines to correlate events across the same source IP or user.
AlertingAlerts are deduplicated, severity-rated, and written to a JSONL audit trail. MITRE ATT&CK tactics and techniques are tagged on each alert.
InterfacesLive curses terminal dashboard or Flask web dashboard with Server-Sent Events streaming.
Threat IntelIP blocklist lookup on every inbound event.
GeoIPOptional GeoIP2 enrichment if a MaxMind database is present.

What ResponSec360 is not

ResponSec360 is not a replacement for production SIEM platforms. It has no persistent storage layer, no multi-node architecture, no role-based access control, and no compliance reporting. It is a detection engineering learning tool and portfolio project.

Replay Mode

Replay mode reads log files from the beginning rather than tailing from the end. This is the primary way to test detection rules against known scenarios.

# Instant replay (no inter-event delay)
$ python main.py cli --replay --speed 0

# Real-time replay (preserves original log timestamps)
$ python main.py cli --replay --speed 1.0

# 20x accelerated replay
$ python main.py cli --replay --speed 20.0

How replay works

The replay engine:

  1. Reads all configured log files completely
  2. Parses every line into LogEvent objects
  3. Sorts all events chronologically by their log timestamps
  4. Feeds events to the correlation engine with scaled inter-event delays
ℹ

Because replay uses the timestamps embedded in the log lines (not wall clock time), detection results are identical regardless of replay speed. This is critical for regression testing — you get the same alerts at speed 0 as at speed 1.0.

Using replay for testing new rules

When you write a new detection rule, the workflow is:

  1. Add sample log entries that should trigger the rule to a file in sample_logs/
  2. Run python main.py cli --replay --speed 0
  3. Verify the alert fires with the correct severity and details
  4. Add entries that should not trigger (false positive testing)
  5. Adjust thresholds in config.yaml until the behavior is correct

Detection Rules

Each rule is a Python class in correlation/engine.py that inherits from BaseRule. Rules maintain their own time-windowed state and fire alerts via the AlertManager when thresholds are crossed.

ssh_brute_force_success HIGH T1110.001 — Brute Force: Password Guessing

Multiple SSH login failures from an IP address followed by a successful SSH login from the same IP within the time window. This is the most reliable indicator of a successful credential attack — the attacker completed a brute force and gained access.

failure_threshold: 5
time_window_seconds: 300

failed_attempts, successful_user, window_seconds

A legitimate user with multiple devices or a misconfigured SSH client repeatedly failing before succeeding. Whitelist known admin IPs or raise failure_threshold.

ssh_brute_force MEDIUM T1110.001 — Brute Force: Password Guessing

A high volume of SSH failure events from a single IP regardless of whether a successful login occurs. Catches bots and scanners that are spraying credentials without success. This rule fires independently of rule 1 — if the attacker eventually succeeds, both rules will fire.

failure_threshold: 10
time_window_seconds: 60

After firing, the alert is suppressed for time_window_seconds for the same IP. Resets on each window.

multi_service_enum MEDIUM T1046 — Network Service Discovery

A single IP interacting with multiple distinct services within a short window. For example, an IP that touches SSH, then HTTP, then triggers a firewall block — classic reconnaissance behavior. The rule counts unique service names, not total events.

service_threshold: 3
time_window_seconds: 120

This rule correlates across all log sources simultaneously — an IP in auth.log and apache_access.log will contribute to the same window.

web_scanner MEDIUM T1595.003 — Active Scanning: Wordlist Scanning

A high rate of HTTP 4xx errors (404, 403, 401, 400) from a single IP. Tools like DirBuster, Gobuster, and ffuf produce this pattern when brute-forcing directories and files. The rule triggers on error volume, not on user-agent, making it harder to evade.

error_threshold: 20
time_window_seconds: 60

Includes a sample_urls list of up to 10 of the attempted paths.

web_attack HIGH T1190 — Exploit Public-Facing Application

Signature-based matching against HTTP request URLs and User-Agent strings. Fires immediately on the first matching request — no threshold required. Patterns cover SQL injection, XSS, path traversal, command injection, and common webshell probe strings.

union select, ' or 1=1, <script>, ../../../, cmd.exe, /etc/passwd, base64_decode, eval(, exec(, wget http, curl http

Extend the patterns list in config.yaml under web_attack. All matching is case-insensitive and checked against the decoded URL.

priv_escalation HIGH T1548.003 — Abuse Elevation: Sudo and Su

Repeated sudo or su failures from the same user. Indicates an attacker who has achieved a low-privilege shell and is attempting to escalate to root. The rule resets its window after firing to avoid infinite suppression.

failure_threshold: 3
time_window_seconds: 300

Username (not IP) since sudo/su are local operations without an obvious source IP.

port_scan MEDIUM T1046 — Network Service Discovery

A single IP reaching many distinct destination ports within a short window, as recorded in UFW or iptables BLOCK entries in syslog. This is the classic Nmap/Masscan signature. The rule counts distinct ports, not total blocked packets.

port_threshold: 10
time_window_seconds: 60

Includes sorted list of up to 30 scanned ports for investigation.

user_creation LOW T1136.001 — Create Account: Local Account

Any new local user account creation (useradd events in auth.log). Severity is low but importance is high — attackers with root access frequently create backdoor accounts for persistence. Fires on every creation without a threshold.

New username, hostname. Correlate with nearby ssh_brute_force_success alerts to determine if this followed a compromise.

If your environment has automated provisioning, whitelist the provisioning system's hostname or change severity to low.

suspicious_useragent MEDIUM T1595 — Active Scanning

HTTP requests containing user-agent strings associated with known attack tools. This is an allowlist-inversion approach: legitimate browsers never send sqlmap or nikto as their user-agent. Fires immediately on first match.

sqlmap, nikto, nmap, masscan, zgrab, dirbuster, gobuster, wfuzz, hydra, metasploit

Sophisticated attackers spoof their user-agent. This rule catches lazy/automated tools. Use in combination with behavioral rules for better coverage.

threat_intel_hit HIGH T1133 — External Remote Services

Any activity from an IP address listed in the threat intelligence blocklist (data/known_bad_ips.txt). Fires on the first event from a known-bad IP, then suppresses repeat alerts for the same IP for 5 minutes to avoid flooding.

Add IPs one per line to data/known_bad_ips.txt. Lines starting with # are comments. Reload requires restart.

Sets is_known_bad: true on the alert, which is highlighted in both dashboards.

Sliding Window Mechanics

The SlidingWindow class in correlation/engine.py is the core data structure used by most detection rules to implement time-based correlation.

How it works

Each rule instance maintains a dictionary of sliding windows, keyed by source IP (or username). When a new event arrives, the window:

  1. Appends the event timestamp and metadata (event ID, service name, port, etc.)
  2. Purges events older than window_seconds relative to the newest event in the deque
  3. Exposes count() and get_distinct(key_fn) for threshold checking
ℹ

Key design decision: Purging is based on the newest event timestamp in the window, not datetime.utcnow(). This ensures replay mode (with historical timestamps) produces identical results to live mode. An event from 2024 stays in window relative to other 2024 events, not relative to the current year.

Thread safety

Each SlidingWindow instance has its own threading.Lock. The correlation engine calls rules from a worker thread pool, so all state mutations are protected.

Memory management

A background maintenance thread runs every 30 seconds (configurable) to clean up expired deduplication cache entries in the AlertManager. Individual SlidingWindow instances self-clean on each access.

False Positive Tuning

Every detection rule has knobs to reduce false positives. The general principle is: start with tight thresholds, observe your environment for a week, then relax where needed.

IP whitelisting

Rules that support false_positive_whitelist skip processing for those IPs entirely:

# config.yaml
ssh_brute_force_success:
  false_positive_whitelist:
    - "10.0.0.5"    # admin jump host
    - "192.168.1.1"  # internal monitoring

Threshold adjustment

For environments with aggressive legitimate tooling (e.g., internal vulnerability scanners), raise thresholds:

web_scanner:
  error_threshold: 50    # was 20
  time_window_seconds: 30  # tighter window

Disabling rules

To disable a rule entirely:

multi_service_enum:
  enabled: false

Deduplication window

If alerts are too noisy from persistent scanners, increase the dedup window:

alerting:
  dedup_window_seconds: 300  # 5 minutes instead of 60

Adding a New Log Source

Adding support for a new log format is a three-step process. No changes are required to the core engine or alerting layers.

1

Create a parser class

Create parsing/myformat_parser.py. Implement a parse(line: str) -> Optional[LogEvent] method:

# parsing/myformat_parser.py
import re
from typing import Optional
from parsing.event import LogEvent
from utils.sanitizer import sanitize_ip

RE = re.compile(r"^(?P<ip>[\d.]+)\s+(?P<action>\S+)\s+(?P<user>\S+)$")

class MyFormatParser:
    def __init__(self, source_name="", source_file=""):
        self.source_name = source_name
        self.source_file = source_file

    def parse(self, line: str) -> Optional[LogEvent]:
        m = RE.match(line)
        if not m:
            return None
        ip = sanitize_ip(m.group("ip"))
        if not ip:
            return None
        return LogEvent(
            source_name=self.source_name,
            source_type="myformat",
            src_ip=ip,
            username=m.group("user"),
            event_category="auth",
            event_action=m.group("action"),
            event_outcome="unknown",
            raw=line,
        )
2

Register the parser

Add one line to parsing/parser_registry.py:

from parsing.myformat_parser import MyFormatParser

PARSER_REGISTRY: Dict[str, Type] = {
    "auth":         AuthLogParser,
    "apache_access": ApacheAccessParser,
    "syslog":        SyslogParser,
    "myformat":      MyFormatParser,  # ← add this
}
3

Add to config.yaml

log_sources:
  - name: "my_custom_log"
    path: "/var/log/myapp/app.log"
    type: "myformat"
    enabled: true

Writing a New Detection Rule

All detection rules are subclasses of BaseRule in correlation/engine.py. Here is a minimal example that detects repeated login failures for a specific username (e.g., root):

class RootLoginAttemptRule(BaseRule):
    """
    MEDIUM severity.
    Detects any SSH login attempt against the root account.
    High-value target — should rarely if ever see direct root logins.
    """

    name = "root_login_attempt"
    description = "Direct root login attempted via SSH"
    mitre_tactic = "Credential Access"
    mitre_technique = "T1078.003 - Valid Accounts: Local Accounts"

    def __init__(self, config, alert_manager):
        super().__init__(config, alert_manager)
        self._seen: Dict[str, SlidingWindow] = defaultdict(
            lambda: SlidingWindow(self._window_seconds())
        )

    def process(self, event: LogEvent):
        # Only care about SSH auth failures targeting root
        if event.service != "ssh":
            return
        if event.username != "root":
            return
        if not event.src_ip:
            return

        ip = event.src_ip
        self._seen[ip].add(event.timestamp, metadata=event.event_id)
        count = self._seen[ip].count()

        if count >= self._threshold("attempt_threshold", default=3):
            self._fire(Alert(
                rule_name=self.name,
                rule_description=self.description,
                severity=self._severity(),
                timestamp=event.timestamp,
                src_ip=ip,
                username="root",
                event_count=count,
                affected_services=["ssh"],
                triggering_events=[m for _, m in self._seen[ip].get_all()],
                mitre_tactic=self.mitre_tactic,
                mitre_technique=self.mitre_technique,
                details={"attempt_count": count},
            ))
            # Reset window to avoid re-firing on every subsequent attempt
            self._seen[ip] = SlidingWindow(self._window_seconds())

Then add it to the CorrelationEngine.__init__ rule list:

self._rules: List[BaseRule] = [
    SSHBruteForceSuccessRule(config, alert_manager),
    # ... other rules ...
    RootLoginAttemptRule(config, alert_manager),  # ← add here
]

Add its config block to config.yaml:

root_login_attempt:
  enabled: true
  attempt_threshold: 3
  time_window_seconds: 60
  severity: "medium"

Querying alerts with jq

# Show all HIGH alerts
$ jq 'select(.severity == "high")' logs/alerts.jsonl

# Count alerts by rule
$ jq -r '.rule_name' logs/alerts.jsonl | sort | uniq -c | sort -rn

# Find all IPs that triggered alerts
$ jq -r '.src_ip // empty' logs/alerts.jsonl | sort -u

# Find brute force + success alerts from known-bad IPs
$ jq 'select(.rule_name == "ssh_brute_force_success" and .is_known_bad == true)' \
  logs/alerts.jsonl

# Show alerts in the last hour (requires date math)
$ jq --arg ts "$(date -u -Iseconds)" \
  'select(.timestamp > ($ts | ltrimstr("T") | .[0:10]))' logs/alerts.jsonl

Audit Trail

ResponSec360 maintains two separate log streams:

FileFormatContents
logs/alerts.jsonlJSONL (one alert per line)All fired security alerts with full context, timestamps, and triggering event IDs. This is the primary audit trail.
logs/responsesec360.logPlain text (rotating)Operational logs: startup, errors, parse failures, maintenance events. Not security alerts. Rotates at 10MB, keeps 5 files.

The triggering_events list in each alert contains the event_id UUIDs of the specific log events that contributed to the alert. These UUIDs are generated at parse time and stored in the alert for traceability — in a production system you would index events by ID for lookup.

Security Notes

Log poisoning prevention

ResponSec360 applies several defenses against log injection attacks where an adversary crafts malicious log lines to trick the parser or dashboard:

DefenseImplementation
IP validationAll IP fields are passed through ipaddress.ip_address(). Anything that isn't a valid IPv4 or IPv6 address (including SQL fragments, ANSI escapes, or extra characters) is rejected and the line is dropped.
ANSI stripTerminal escape sequences are removed from all fields before display. Prevents dashboard injection via crafted log lines.
Line length capLines over 8,192 bytes are dropped at ingestion. Prevents memory exhaustion from crafted log lines.
Null byte removalNull bytes are stripped from all string fields.
Field truncationString fields are truncated at type-appropriate lengths (64 bytes for usernames, 2048 for URLs, etc.).
Exception isolationEach rule's process() call is wrapped in try/except. A crashing rule does not stop other rules from processing.

Process security

Run ResponSec360 as a dedicated low-privilege user with read-only access to log files and write access only to the logs/ directory. Never run it as root.

# Create a dedicated service user
$ sudo useradd -r -s /bin/false -d /opt/responsesec360 responsesec360
$ sudo chown -R responsesec360:responsesec360 /opt/responsesec360
$ sudo usermod -aG adm responsesec360  # read access to /var/log

Dashboard opens but no alerts appear Running in live tail mode with no recent activity Use --replay flag to process existing log content. Verify log paths in config.yaml are correct and readable.
FileNotFoundError: config.yaml Running main.py from wrong directory Run from the siem/ directory: cd siem && python main.py cli
